Chapter 1 — Compute

EC2, Auto Scaling, Lambda, ECS/Fargate/EKS, Elastic Beanstalk — the backbone of nearly every SAA scenario question.

EC2 ★★★★★

1. What AWS Loves Asking

  • Instance purchasing options and when each is appropriate (On-Demand, Reserved, Spot, Savings Plans, Dedicated Hosts/Instances)
  • Placement groups (Cluster vs Spread vs Partition) and their failure/latency tradeoffs
  • EBS volume types and matching IOPS/throughput needs to workload
  • Instance metadata service (IMDSv1 vs IMDSv2) and SSRF protection
  • Hibernate vs Stop vs Terminate behavior
  • Spot Instance interruption handling and Spot Fleet allocation strategies
  • Enhanced networking / EFA for HPC workloads
  • Dedicated Host vs Dedicated Instance (licensing scenarios)

2. Must Know Facts

✓ Must Know

IMDSv2 requires a session token (PUT request first) — mitigates SSRF attacks that could steal IAM role credentials via IMDSv1. Enforce via: HttpTokens = required.

✓ Must Know — Stop vs Terminate vs Hibernate
  • Stop: EBS-backed root volume persists; instance store volumes are LOST.
  • Terminate: Root EBS volume deleted by default UNLESS "Delete on Termination" is unchecked.
  • Hibernate: RAM contents saved to EBS root volume — faster boot, preserves in-memory state. Not supported on instance-store-backed instances.
✓ Must Know — Placement Groups
  • Cluster — single AZ, low latency/high throughput (HPC, tightly coupled). Risk: correlated hardware failure.
  • Spread — each instance on distinct underlying hardware, max 7 instances per AZ per group. For critical small-count apps.
  • Partition — instances spread across logical partitions (racks), each partition has own network/power. For HDFS, Kafka, Cassandra.

3. Common Exam Traps

⚠ Exam Trap

Spot Instance interruption notice is only a 2-minute warning, NOT something you can prevent — design for interruption tolerance, don't try to "reserve" Spot capacity. The EXAM wants you to say workload must be fault-tolerant/stateless.

⚠ Exam Trap

EC2 Auto Recovery does NOT help with Availability Zone failure — it only recovers from underlying HARDWARE failure within the same AZ. If the question says "entire AZ is down," the answer is Multi-AZ architecture / ASG across AZs, NOT EC2 auto recovery.

Auto Scaling Groups (ASG) ★★★★★

1. What AWS Loves Asking

  • Scaling policies: Target Tracking vs Step vs Simple vs Scheduled vs Predictive Scaling
  • Cooldown periods and how they interact with multiple policies
  • Lifecycle hooks (Pending:Wait / Terminating:Wait) for custom actions
  • Health checks: EC2 status checks vs ELB health checks — must enable ELB explicitly
  • Warm pools to reduce launch latency
  • Termination policies (which instance gets killed first)

2. Must Know Facts

✓ Must Know

By default, ASG uses only EC2 status checks for health — it will NOT replace an instance failing ELB health checks unless you explicitly enable "ELB health check type." This is a classic trap: "instances are unhealthy at app layer but ASG isn't replacing them" → enable ELB health checks on the ASG.

✓ Must Know — Lifecycle Hooks

Lifecycle hooks pause an instance in Pending:Wait or Terminating:Wait to run custom scripts (e.g., pull logs before termination, install software before going into service). Default timeout per hook: 1 hour. Max: 48 hours.

3. Common Exam Traps

⚠ Exam Trap

Scaling Cooldown prevents additional scaling activities from the SAME policy too soon — it does NOT block all scaling from all policies. Target Tracking largely ignores cooldown and has its own scale-in/out stabilization logic.

⚠ Exam Trap

Default termination policy order: AZ with most instances → oldest launch configuration/template → closest to next billing hour. To PRESERVE specific instances → use Instance Protection.

AWS Lambda ★★★★★

1. What AWS Loves Asking

  • Concurrency: reserved vs provisioned concurrency, and cold start mitigation
  • Lambda + VPC — ENI attachment, NAT Gateway needed for internet access
  • Event source mappings: poll-based (SQS, Kinesis, DynamoDB Streams) vs push-based (S3, SNS, API Gateway)
  • Synchronous vs asynchronous invocation and retry behavior
  • Destinations vs DLQ for async failures
  • Lambda timeout, memory, and CPU relationship
  • Lambda@Edge vs CloudFront Functions

2. Must Know Facts

✓ Must Know — Key Limits
  • Maximum execution timeout: 15 minutes (900 seconds)
  • Memory range: 128 MB – 10,240 MB. CPU scales proportionally with memory — more memory = more vCPU.
  • Deployment package: 50 MB zipped, 250 MB unzipped (incl. layers). Container images up to 10 GB.
  • /tmp ephemeral storage: up to 10,240 MB (default 512 MB)
✓ Must Know — VPC Lambda

Lambda functions in a VPC need a NAT Gateway (or VPC endpoint) to reach the internet or other AWS services. A VPC-attached Lambda has NO internet access by default — it only has access to resources within the VPC.

✓ Must Know — Concurrency
  • Reserved Concurrency — caps max concurrent executions for a function; also guarantees that capacity from the account pool
  • Provisioned Concurrency — pre-initializes execution environments to eliminate cold starts; costs extra even when idle

3. Common Exam Traps

⚠ Exam Trap — Async vs DLQ vs Destinations

Async invocations (S3, SNS, EventBridge) auto-retry 2 times and can route failed events to a DLQ (SQS or SNS only) OR a Destination (SQS, SNS, Lambda, EventBridge — newer, AWS-recommended). Destinations also capture success events.

For SQS poll-based sources, the correct failure mechanism is a redrive policy + DLQ on the SQS queue itself (maxReceiveCount), NOT Lambda Destinations.

⚠ Exam Trap — Memory ≠ Timeout

Increasing Lambda memory increases CPU and network throughput proportionally. If a CPU-bound function is slow, the exam answer is often "increase memory allocation," NOT "increase timeout." Increasing timeout alone doesn't fix a CPU bottleneck.

5. Why the Other Options are Wrong

An SQS-triggered Lambda fails repeatedly on certain messages, blocking the queue. What is the best solution?
✓ Correct: Configure a redrive policy on the SQS queue with a DLQ and maxReceiveCount
✗ Lambda Destination for failures — Destinations apply to asynchronous invocations, not SQS poll-based event source mappings.
✗ Increase Lambda timeout — Doesn't address poison-pill messages; the message will simply keep failing and retrying.
✗ Increase Reserved Concurrency — Doesn't solve the underlying issue of repeatedly failing messages blocking the queue.

6. Memory Tricks

🧠 Memory Trick

"Memory buys everything" — in Lambda, more memory = more CPU = more network = faster execution. One slider controls all three.

"Provisioned = Pre-Provisioned" — Provisioned Concurrency literally pre-runs the init code so the cold start never happens for that capacity.

Chapter 2 — Storage

S3, EBS, EFS, FSx, Storage Gateway, Snow Family — storage selection is one of the highest-yield decision categories on the exam.

Amazon S3 ★★★★★

1. What AWS Loves Asking

  • Storage class selection based on access pattern and retrieval time/cost
  • Lifecycle policies (transition + expiration rules) and minimum storage duration billing
  • S3 versioning + MFA Delete + Object Lock (compliance vs governance mode)
  • Encryption options: SSE-S3 vs SSE-KMS vs SSE-C vs client-side
  • Replication: CRR vs SRR, requirements (versioning must be enabled on both buckets)
  • Presigned URLs vs CloudFront Signed URLs vs Signed Cookies
  • S3 Select / Glacier Select for partial object retrieval

2. Must Know Facts

✓ Must Know — Consistency

S3 provides STRONG read-after-write consistency for ALL operations (PUTs and DELETEs, including overwrite PUTs). Older study material referencing "eventual consistency on overwrites" is outdated.

✓ Must Know — Object Lock Modes
  • Governance — users with special IAM permission (s3:BypassGovernanceRetention) CAN override/delete.
  • Compliance — NO ONE, including root, can delete or overwrite until retention expires. Cannot be shortened or removed even by AWS.
✓ Must Know — Replication Requirements

Replication (CRR/SRR): Versioning MUST be enabled on BOTH source and destination buckets. Replication is NOT retroactive — only applies to objects created AFTER replication is enabled (use S3 Batch Replication for existing objects).

✓ Must Know — Upload Limits
  • Multipart upload: REQUIRED for objects >5 GB, recommended for >100 MB
  • Maximum single PUT object size: 5 GB
  • Maximum object size overall: 5 TB (via multipart)

2. Common Exam Traps

⚠ Exam Trap

st1/sc1 (HDD-based) CANNOT be used as boot volumes — only SSD types (gp2/gp3/io1/io2) can boot an instance.

⚠ Exam Trap — Multi-Attach

Multi-Attach allows the SAME io1/io2 volume to attach to MULTIPLE EC2 instances, but ONLY within the same AZ, and the application/filesystem must be cluster-aware. A standard filesystem like ext4 will corrupt with concurrent writes from multiple instances.

✓ Must Know — gp3 vs gp2

gp3 decouples IOPS/throughput from storage size (baseline 3,000 IOPS / 125 MB/s regardless of size). Unlike gp2, where IOPS scales WITH volume size (3 IOPS per GB, burstable via I/O credit bucket).

Amazon RDS ★★★★★

1. What AWS Loves Asking

  • Multi-AZ vs Read Replicas — purpose, failover, sync vs async replication
  • Enhanced Monitoring vs Performance Insights
  • Automated backups, manual snapshots, PITR window (up to 35 days)
  • RDS Proxy for connection pooling / failover speed
  • Cross-region Read Replicas for geographic read scaling + DR
  • Encryption: must be enabled at creation, can't retroactively encrypt directly

2. Must Know Facts

✓ Must Know — Multi-AZ vs Read Replica
  • Multi-AZ: SYNCHRONOUS replication to a standby. Standby is NOT readable. Purpose: HIGH AVAILABILITY / automatic failover (~60-120 sec).
  • Read Replica: ASYNCHRONOUS replication. Replica IS readable. Purpose: READ SCALING. Can be cross-region. Must be manually promoted if primary fails.
✓ Must Know — Enhanced Monitoring vs Performance Insights
  • Enhanced Monitoring = OS-level metrics (processes, kernel threads, RDS child processes). Granularity down to 1 second. Think: OS.
  • Performance Insights = Database/SQL-level (top queries, wait states, Average Active Sessions). Think: SQL.
  • CPUUtilization, FreeableMemory, DatabaseConnections are STANDARD CloudWatch metrics, NOT Enhanced Monitoring exclusive metrics.

3. Common Exam Traps

⚠ Exam Trap — Read Replica ≠ High Availability

Students confuse these because both involve a "second copy." Multi-AZ = automatic failover to standby (HA). Read Replica = manual promotion only, used for reads (scaling). The standby in Multi-AZ is not readable. The Read Replica is readable but doesn't auto-failover.

⚠ Exam Trap — Retroactive Encryption

Cannot enable encryption on an existing unencrypted RDS instance directly. Process: 1) take a snapshot → 2) copy the snapshot WITH encryption enabled → 3) restore a new instance from the encrypted snapshot copy.

⚠ Exam Trap — Cross-Region Read Replica Purpose

Cross-region read replicas serve BOTH read scaling for global users AND act as a DR target (promote in a DR region). The trap: questions framing "reduce read latency for global users" often distract with "Multi-AZ" — but Multi-AZ is single-region and the standby isn't even readable.

4. Easy Confusions

1. Must Know Facts

✓ Must Know — RCU/WCU Math

AWS KMS ★★★★★

1. Must Know Facts

✓ Must Know — Envelope Encryption

KMS does NOT encrypt data larger than 4 KB directly. For larger data, KMS uses ENVELOPE ENCRYPTION: generates a unique data key → encrypts your data locally with the data key (AES-256) → encrypts the data key itself with the CMK (the "envelope"). Only the encrypted data key + encrypted data are stored; KMS never processes your actual large dataset.

✓ Must Know — Key Types
  • AWS Owned Keys — not in your account, shared, no visibility, no extra KMS cost (used by default SSE-S3)
  • AWS Managed Keys (aws/service-name) — in your account, auto-rotated yearly, visible but not manageable
  • Customer Managed Keys (CMK) — full control: rotation, policy, enable/disable, deletion (7-30 day waiting period), aliasing, cross-account sharing

2. Common Exam Traps

⚠ Exam Trap — CMK Deletion

Deleting a KMS CMK is intentionally hard — mandatory waiting period of 7-30 days before permanent deletion. All data encrypted under that key becomes permanently unrecoverable once deleted.

⚠ Exam Trap — Multi-Region Keys

A standard single-region CMK CANNOT be used directly to decrypt in another region. Cross-region encrypted data requires KMS Multi-Region Keys (replicated key with same key material, but separate key IDs per region).

2. Must Know Facts

✓ Must Know — Standard vs FIFO SQS
  • FIFO: Strict ordering + exactly-once processing. Throughput: 300 msg/sec (no batching) / 3,000 msg/sec (with batching)
  • Standard: Best-effort ordering (NOT guaranteed), at-least-once delivery (possible duplicates), virtually UNLIMITED throughput
✓ Must Know — Visibility Timeout

Once a consumer receives an SQS message, it becomes invisible to other consumers for the visibility timeout duration (default 30 sec, max 12 hours). If not deleted before timeout expires, the message becomes visible again and may be redelivered — if this happens too often, it lands in a DLQ if configured.

3. Common Exam Traps

⚠ Exam Trap — SNS Fan-Out Pattern

SNS → multiple SQS queues = fan-out with durability. If one consumer's system is down, messages queue up in SQS and are processed when it recovers. SNS pushing directly to HTTP endpoints has no such buffering. "Multiple systems must independently process the same event, even if one is temporarily down" → SNS + SQS fan-out.

⚠ Exam Trap — Long Polling

Long Polling (WaitTimeSeconds up to 20s) reduces empty responses and API call COST. If the question says "reduce costs of polling an SQS queue" → enable Long Polling, NOT "increase visibility timeout" (completely different concern).

⚠ Exam Trap — Standard Queue Duplicates

Standard queue "at-least-once" means your consumer logic MUST be idempotent. If strict no-duplicate processing is required → FIFO queue with deduplication.

4. Why the Other Options are Wrong

An order event must trigger billing, shipping, and analytics independently, each processing at their own pace, with guaranteed delivery even if one is temporarily down.
✓ Correct: SNS topic with 3 subscribed SQS queues (fan-out)
✗ SNS with 3 HTTP endpoint subscriptions — No built-in durability/retry buffering; if a system is down, notification can be lost.
✗ Single SQS queue consumed by all 3 systems — A standard SQS message is deleted once processed by ONE consumer; the other two would never see it.

⚠ Exam Trap — IAM Is Global

IAM is GLOBAL — a common trap assumes you need to recreate IAM roles/users per region. You don't. The same IAM role can be used by resources in any region.

⚠ Exam Trap — S3 Bucket Scope
